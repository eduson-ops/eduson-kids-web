02 / ПРОБЛЕМА
Знакомая боль

Почему uvicorn умирает каждую ночь?

✕ Как было
  • Запустил в SSH → закрыл ноут → сервер умер

05 / SYSTEMD
Начальник всех процессов

systemd — PID 1 в Ubuntu.

Первый процесс, который стартует при загрузке системы. Все остальные — его дети. Ты пишешь юнит-файл с декларацией «что нужно», systemd делает «как».

→

Что systemd умеет

  • Запускать процессы в правильном порядке
  • Перезапускать при падении
  • Собирать stdout/stderr в journal
  • Передавать EnvironmentFile
  • Ограничивать CPU/RAM (через cgroups)

06 / СТРУКТУРА
Три секции манифеста

Анатомия .service-файла.

[UNIT]

Метаданные

Описание, зависимости от других юнитов, порядок загрузки. After=network.target — «стартуй после сети».

[SERVICE]

Как запускать

От какого юзера, в какой папке, какую команду. Restart=always — «поднимай, если упало».

[INSTALL]

Автозапуск

WantedBy=multi-user.target — «стартуй при обычной загрузке ОС».

Минимальный шаблон

[Unit]
Description=Моё описание
After=network.target

[Service]
User=ubuntu
WorkingDirectory=/path/to/project
ExecStart=/path/to/venv/bin/uvicorn main:app --host 0.0.0.0 --port 8000
Restart=always

[Install]
WantedBy=multi-user.target
ini

09 / ПРАКТИКА · ШАГ 3
Жизненный цикл сервиса

03 — systemctl: 6 команд на 90%

01 / DAEMON-RELOAD

sudo systemctl daemon-reload

После правки .service — ОБЯЗАТЕЛЬНО. Иначе systemd не видит изменения.

02 / ENABLE

sudo systemctl enable vibe-api

Автозапуск при ребуте. Создаёт symlink в multi-user.target.wants/.

03 / START

sudo systemctl start vibe-api

Запустить прямо сейчас. Не ждать ребута.

04 / STATUS

sudo systemctl status vibe-api

Зелёный active (running) — победа. Красный failed — идём читать логи.

05 / RESTART

sudo systemctl restart vibe-api

После git pull или правок в коде — рестарт. 2–3 секунды даунтайма.

06 / STOP

sudo systemctl stop vibe-api

Временно выключить. Без disable — при ребуте поднимется обратно.

15 / ТИПИЧНЫЕ ОШИБКИ
80% затыков на уроке

Три грабли, на которые наступят все

failed

Status: failed, код 217

Причина: в юните User= указан несуществующий пользователь.

Фикс: whoami на VM — и вставить это в User=. Обычно ubuntu.

exec format

Путь до uvicorn неправильный

Причина: ExecStart=uvicorn ... — systemd не знает $PATH.

Фикс: which uvicorn внутри venv → вставить полный путь /home/ubuntu/.../.venv/bin/uvicorn.

KeyError

.env не подхватывается

Причина: формат EnvironmentFile строгий. Никаких пробелов и кавычек.

Фикс: KEY=value, одно на строку. Без export, без " ".

Правило отладки

  1. systemctl status vibe-api — что говорит?
  2. journalctl -u vibe-api -n 50 — последние 50 строк
  3. Если ничего непонятно — journalctl -xe (расширенный вывод)

19 / БОНУС
Для тех, кто успел раньше

Logrotate — не съесть диск.

Без ротации лог растёт бесконечно. Неделя — 20 GB, диск кончился, сервер лёг. Logrotate режет логи по расписанию и gzip-ит старые.

/etc/logrotate.d/vibe-api

/var/log/vibe-api.log {
    daily
    rotate 7
    compress
    missingok
    notifempty
    copytruncate
}
bash
  • daily — резать раз в сутки
  • rotate 7 — хранить 7 старых версий
  • compress — gzip-ить архивы
  • copytruncate — не нужно рестартить сервис

Тест работы

$ sudo logrotate -d /etc/logrotate.d/vibe-api
считаем вывод — что бы сделал logrotate
$ sudo logrotate -f /etc/logrotate.d/vibe-api
— форсированный прогон сейчас
$ ls /var/log/ | grep vibe
vibe-api.log
vibe-api.log.1.gz

предыстория К уроку 10 этот конфиг уже будет у тебя — там мы подключим slowapi и структурированные логи. Ты в теме раньше остальных.
