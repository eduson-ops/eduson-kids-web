анатомия запроса 04 / 22

Три роли: system · user · assistant

  • system — инструкция сотруднику. Кто ты, что делаешь, чего не делаешь.
  • user — то, что пишет пользователь. Конкретный запрос, входные данные.
  • assistant — то, что ответил LLM. При few-shot сюда кладём примеры ожидаемого ответа.
Правило: не смешивай инструкции и данные. Инструкция → system. Данные → user. Никогда наоборот.

параметры LLM 05 / 22

Температура и max_tokens

0.0–0.3 предсказуемо

Переводы, факты, классификация, парсинг. Почти детерминированно, близко к «тот же ответ каждый раз».

0.6–0.9 золотая середина

Генерация текста, гороскопы, объяснения, посты. Разнообразно, но управляемо.

1.0+ творчество + галлюцинации

Стихи, безумные идеи, brainstorming. Готовься к тому, что модель начнёт выдумывать факты.

max_tokens — потолок длины

1 токен ≈ 2–3 русских символа. Для гороскопа 120–150 слов ставим max_tokens=300. Без этого модель может писать роман — и ты заплатишь за каждый токен.

Правило большого пальца

Если модель галлюцинирует — сначала ужесточи промпт, потом понижай температуру. Не наоборот.

Speaker notes

Живьём покажите разницу: один и тот же промпт с temp=0.2 и temp=1.1 на три вызова каждый. Стабильность vs хаос — наглядно.

Цифры из мира денег: максимум токенов = максимум счёта. У GigaChat 1M токенов ≈ 30 ₽. Если юзер дёрнет API 10k раз без лимита на max_tokens — легко улетит 300 ₽ из кармана.

промпт-техника 06 / 22

Few-shot: покажи 1–2 примера вместо 100 слов инструкций

Когда модели трудно понять формат по описанию — дай ей образец. 1 пример правильного ответа часто поднимает качество сильнее, чем три абзаца уточнений.

messages[] с примером
[
  {"role": "system",
   "content": "Ты астролог. 120–150 слов, ровно 1 абзац, шутка в конце."},

  {"role": "user",
   "content": "Знак: Овен."},
  {"role": "assistant",
   "content": "Сегодня Овен чувствует прилив энергии. Утро подарит ясность в планах, полдень — новое знакомство, вечер — тихий ужин. Важно: никаких резких решений после 19:00. Кстати, если купишь кактус — он тебя обязательно переживёт."},

  {"role": "user",
   "content": "Знак: Лев."}
]
json
  • Формат скопируется. Модель увидит «один абзац, 120–150 слов, шутка в конце» и повторит структуру.
  • Тон скопируется. Живой, дружелюбный, с мягким юмором — именно таким и будет Лев.
  • Длина скопируется. Если пример 135 слов, ответ будет в районе 120–150.
  • Цена примера ~200 токенов. Это дёшево. Стабильность стоит дороже.
Правило: 1 пример почти всегда лучше 0. 2 примера лучше 1. Больше 3 — редко оправдано: растёт счёт, падает скорость.
Speaker notes

Объясните, почему это работает: LLM — это машина, предсказывающая продолжение. Покажи продолжение — она продолжит в том же стиле.

Практическая ловушка: не делай пример слишком «богатым» (с метафорами, сложным синтаксисом). Модель скопирует и это, и ответ на «простой» знак получится перегруженным.

api/horoscope.py — часть 1

import time import httpx from prompts import ( SYSTEM_PROMPT, FEW_SHOT_USER, FEW_SHOT_ASSISTANT, VALID_SIGNS, ) from gigachat_client import get_gigachat_token # с урока 4 async def _call_gigachat(sign_ru: str) -> str: token = await get_gigachat_token() payload = { "model": "GigaChat", "messages": [ {"role": "system", "content": SYSTEM_PROMPT}, {"role": "user", "content": FEW_SHOT_USER}, {"role": "assistant", "content": FEW_SHOT_ASSISTANT}, {"role": "user", "content": f"Знак: {sign_ru}."}, ], "temperature": 0.8, "max_tokens": 300, } async with httpx.AsyncClient(timeout=20, verify=False) as client: r = await client.post( "https://gigachat.devices.sberbank.ru/api/v1/chat/completions", headers={"Authorization": f"Bearer {token}"}, json=payload, ) r.raise_for_status() return r.json()["choices"][0]["message"]["content"]
  • Ровно 4 сообщения: system + few-shot (user/assistant) + реальный user.
  • temperature=0.8 — творческая задача, но контролируемая.
  • max_tokens=300 — потолок длины, страховка от романов.
  • timeout=20 — больше ждать бессмысленно, юзер уйдёт.
  • raise_for_status() — кидаем ошибку, если 4xx/5xx. Обработаем выше.
Speaker notes

Важно: verify=False появляется тут из-за особенностей сертификатов GigaChat в России. В продакшене потом подтянем корневой сертификат Минцифры (отдельная тема). Сегодня — простой путь.

Покажите в Continue.dev: «Объясни, что делает raise_for_status» — пусть AI-пара это объяснит прямо на уроке.

шаг 5 / 6 — склейка 12 / 22

Роутер и обработка краёв

api/horoscope.py — часть 3
Python
SIGN_RU = { "oven": "Овен", "telec": "Телец", "bliznecy": "Близнецы", "rak": "Рак", "lev": "Лев", "deva": "Дева", "vesy": "Весы", "skorpion": "Скорпион", "strelec": "Стрелец", "kozerog": "Козерог", "vodoley": "Водолей", "ryby": "Рыбы", } async def generate_horoscope(sign: str, source: str) -> dict: sign = sign.lower().strip() if sign not in VALID_SIGNS: raise ValueError( f"unknown sign '{sign}'. " f"allowed: {sorted(VALID_SIGNS)}" ) t0 = time.perf_counter() if source == "gigachat": text = await _call_gigachat(SIGN_RU[sign]) else: text = await _call_yandex(SIGN_RU[sign]) ms = int((time.perf_counter() - t0) * 1000) if "ERROR_UNKNOWN_SIGN" in text: raise ValueError("LLM didn't recognize the sign") return {"sign": sign, "source": source, "text": text, "ms": ms}
  • Валидация на входе. ValueError → FastAPI превратит в 400 (см. слайд 8).
  • Нормализация. lower().strip() — «Lev », «LEV», «lev» → одно и то же.
  • Измерение времени. perf_counter() точнее, чем time.time().
  • Двойная защита. Если модель вернула сентинел — всё равно 400, а не битый гороскоп.
  • SIGN_RU — нужен, потому что промпт на русском, а параметр URL — на латинице.
Итог: 9 строк бизнес-логики. Остальное — промпт и интеграции.
Speaker notes

Обсудите, почему словарь SIGN_RU именно здесь, а не в prompts.py: это «переходник» между внешним API (латиница в URL) и внутренним представлением (кириллица в промпте). Место для таких переходников — рядом с логикой вызова.

анти-паттерны 17 / 22

Типичные ошибки джуна на LLM

Как не надо

  • Склеивать промпт через f-string из юзер-инпута без валидации.
  • Перезапускать 10 раз, пока не сработало, — вместо того чтобы уточнить промпт.
  • Оценивать «на глаз», какая модель лучше. Без чисел это кухонный разговор.
  • Не ставить max_tokens — однажды юзер заплатит за «Войну и мир» вместо гороскопа.
  • Хардкодить ключи API в коде.
  • Кидать ключи и токены в логи «для отладки».

Как надо

  • Валидируй входные данные ещё до формирования промпта.
  • Итерируй промпт осознанно: одну переменную за раз, записывай что и зачем поменял.
  • Меряй метрики: длина, время, валидность формата, попадание в критерии.
  • Ставь max_tokens и timeout всегда. Это твоя страховка от разорения.
  • Храни ключи в .env, в логи маскируй (`sk-***4f2a`).
  • Обрабатывай ошибки LLM осмысленно: 401, 429, 5xx — разные сценарии, разные реакции.
