Что мы сделаем за 60 минут

01

Reverse proxy перед FastAPI. Пользователь стучит на порт 80 — Nginx проксирует на 127.0.0.1:8000.
02
Красивый URL
Вместо http://IP:8000/docs откроется http://IP/docs. Без порта, как взрослый сайт.
03
Настроим ufw
Firewall закроет порт 8000 снаружи. Останутся только 22 (SSH), 80 (HTTP), 443 (HTTPS).
04
Проверим
nginx -t, ufw status, открытие http://IP/docs — всё три должны пройти.
зачем reverse proxy

Три причины поставить Nginx

↯
Безопасность
FastAPI не торчит наружу. Все идут через один понятный порт 80. Внутренний :8000 виден только самому серверу.
⌘
Гибкость
Завтра рядом поставим бота или фронтенд. Nginx разведёт /api → FastAPI, /bot → aiogram, / → статика.
⚷
SSL на раз
На уроке 8 certbot сам правит конфиг Nginx и выдаёт HTTPS. Без Nginx это было бы в 5 раз больнее.
Мысль урока: Nginx — это вахтёр в здании. Все гости проходят через него, а не ломятся в квартиры напрямую.

порты как адреса

У каждого порта своя роль

22 SSH для тебя: заходишь на сервер. Открыт снаружи. allow
80 HTTP для пользователей. Тут сидит Nginx. Открыт снаружи. allow
443 HTTPS зашифрованный HTTP. Откроем сейчас — пригодится на уроке 8. allow
8000 FastAPI только для Nginx. Снаружи — закрыт. deny
11434 Ollama / прочее внутренние сервисы — только localhost. deny

Открой http://<IP>/ — увидишь «Welcome to nginx!». Это дефолтная страница — скоро уберём.
шаг 2/7 · анатомия конфига

Блок server и его части

server {
    listen 80;
    server_name _;

    location / {
        proxy_pass http://127.0.0.1:8000;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
    }
}
nginx
  • listen 80 — слушаем HTTP-порт
  • server_name _ — ловим любой домен (пока нет своего)
  • location / — все пути без исключений
  • proxy_pass — куда отдаём запрос внутри
  • X-Real-IP — FastAPI узнает настоящий IP клиента
127.0.0.1 — это loopback, путь к самому себе. Nginx и FastAPI — на одной VM, общаются «внутри».
шаг 3/7 · пишем свой конфиг

Создаём vibe-api

nano · /etc/nginx/sites-available/vibe-api
ubuntu@vm:~$ sudo nano /etc/nginx/sites-available/vibe-api 
server {
    listen 80;
    server_name _;

    location / {
        proxy_pass http://127.0.0.1:8000;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}
nginx
Сохранение в nano: Ctrl+O → Enter. Выход: Ctrl+X.
шаг 4/7 · включаем сайт

Симлинк и удаление default

enable site
# включаем наш конфиг симлинком ubuntu@vm:~$ sudo ln -s /etc/nginx/sites-available/vibe-api \ /etc/nginx/sites-enabled/ # убираем «Welcome to nginx!» заглушку ubuntu@vm:~$ sudo rm /etc/nginx/sites-enabled/default # проверяем синтаксис — это ОБЯЗАТЕЛЬНО ubuntu@vm:~$ sudo nginx -t nginx: the configuration file /etc/nginx/nginx.conf syntax is ok nginx: configuration file /etc/nginx/nginx.conf test is successful # мягко перечитываем конфиг, без обрыва соединений ubuntu@vm:~$ sudo systemctl reload nginx
Правило: всегда nginx -t перед reload. Если не проверил — однажды сайт упадёт посреди ночи.

шаг 6/7 · firewall

Закрываем лишнее через ufw

⚠ Правило № 1
Сначала allow 22, только ПОТОМ enable. Иначе обрубишь себе SSH.
ufw setup
# 1. разрешаем нужные порты ubuntu@vm:~$ sudo ufw allow 22 # SSH — самое важное! ubuntu@vm:~$ sudo ufw allow 80 # HTTP (Nginx) ubuntu@vm:~$ sudo ufw allow 443 # HTTPS (на завтра) # 2. только после этого включаем firewall ubuntu@vm:~$ sudo ufw enable Command may disrupt existing ssh connections. Proceed? (y|n) y Firewall is active and enabled on system startup # 3. смотрим, что разрешено ubuntu@vm:~$ sudo ufw status Status: active 22/tcp ALLOW Anywhere 80/tcp ALLOW Anywhere 443/tcp ALLOW Anywhere

Мягкая перезагрузка

reload, а не restart — важно!
reload
Мягко перечитывает конфиг. Активные соединения не рвутся. Используем всегда.
restart
Убивает Nginx и поднимает заново. Клиенты видят обрыв. Только в безвыходных случаях.

домашнее задание

Health-check на стороне Nginx

Добавить в Nginx-конфиг отдельный location /health, который отвечает 200 "ok" — без проксирования в FastAPI. Полезно для мониторинга: проверяет, что сам веб-сервер жив. 
server {
    listen 80;
    server_name _;

    location /health {
        return 200 "ok";
        add_header Content-Type text/plain;
    }

    location / {
        proxy_pass http://127.0.0.1:8000;
        # ...
    }
}
nginx
Чек-лист
  • /health возвращает 200 "ok"
  • nginx -t зелёный
  • Конфиг закоммичен в deploy/nginx/
  • Скриншот curl http://IP/health в чат группы
Бонус (+1)
Сделать кастомную 404-страницу с ASCII-артом через error_page 404 /404.html;.
критерии успеха

Что должно работать к концу урока

  • API доступен по http://<IP>/docs — без указания порта 8000.
  • Порт 8000 снаружи закрыт — curl -m 5 http://IP:8000 зависает по таймауту.
  • ufw status показывает только 22 / 80 / 443 в ALLOW.
  • nginx -t отвечает «successful» — конфиг валиден.
  • Ты можешь на пальцах объяснить, зачем reverse proxy и что такое 127.0.0.1.
