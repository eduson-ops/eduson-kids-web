06 · Концепция

Что такое rate-limit на пальцах

Правило: «с одного IP не больше N запросов за T времени». Превысил — получи 429 Too Many Requests и остынь.

Клиент → запрос 1
Клиент → запрос 2
...
Клиент → запрос 10 · OK
Клиент → запрос 11 · 429
429 — код «притормози, не успеваю». Клиент должен ждать Retry-After секунд.

Алгоритмы под капотом

АлгоритмСуть
Fixed windowМинутные корзины. Просто, но есть «пики на границе»
Sliding windowСкользящее окно. Ровнее. Это slowapi
Token bucketБанка токенов, каждый запрос — один токен
Leaky bucketОбратный token bucket, плавная выдача
Для учебного проекта sliding window — идеально. Не надо писать руками — есть готовая библиотека slowapi.

08 · Практика · шаг 2

Подключаем Limiter в main.py

Python
from fastapi import FastAPI, Request from slowapi import Limiter, _rate_limit_exceeded_handler from slowapi.util import get_remote_address from slowapi.errors import RateLimitExceeded limiter = Limiter(key_func=get_remote_address) app = FastAPI() app.state.limiter = limiter app.add_exception_handler(RateLimitExceeded, _rate_limit_exceeded_handler) @app.get("/horoscope") @limiter.limit("10/minute") async def horoscope(request: Request, sign: str, source: str = "gigachat"): # ... твой код с LLM-вызовом return {"sign": sign, "text": "..."}
Обязательно первым аргументом идёт request: Request. Иначе slowapi не найдёт, откуда брать IP — молча пропустит лимит.
Разные лимиты на разные эндпоинты: для /ping можно 100/minute, для /horoscope — 10/minute, потому что LLM дорогой.

10 · Логирование

Python logging: забудь про print()

Почему print — зло на проде

  • Нет уровней: не отличишь debug от ошибки
  • Нет формата: просто текст, попробуй распарсь
  • Не ротируется, валится в stdout
  • Невозможно выключить без правки кода

Иерархия уровней

УровеньКогда
DEBUGМелочь для разработки
INFOОбычная жизнь сервиса
WARNINGСтранность, но не упало
ERRORЧто-то сломалось, клиенту плохо
CRITICALСервис умирает прямо сейчас

logging_config.py

Python
import logging from logging.handlers import RotatingFileHandler def setup_logging(): fmt = "%(asctime)s %(levelname)-8s %(name)s %(message)s" formatter = logging.Formatter(fmt) file_h = RotatingFileHandler( "/var/log/vibe-api.log", maxBytes=10_000_000, # 10 MB backupCount=5, ) file_h.setFormatter(formatter) root = logging.getLogger() root.setLevel(logging.INFO) root.addHandler(file_h) log = logging.getLogger("vibe")
11 · Middleware

Middleware, которая логирует каждый запрос

Python
import time from fastapi import Request from logging_config import log @app.middleware("http") async def log_requests(request: Request, call_next): start = time.perf_counter() ip = request.client.host path = request.url.path method = request.method try: response = await call_next(request) elapsed_ms = (time.perf_counter() - start) * 1000 log.info(f"{ip} {method} {path} → {response.status_code} · {elapsed_ms:.0f}ms") return response except Exception as exc: log.error(f"{ip} {method} {path} → 500 · {exc!r}") raise

12 · Ловушка

Почему за Nginx все запросы идут с 127.0.0.1

Проблема

Клиент 95.2.14.7
↓
Nginx :443
↓
FastAPI видит 127.0.0.1
Все пользователи = один IP. Slowapi забанит всех одновременно. Это не rate-limit, это DoS на себя.

Правильный конфиг Nginx

location / {
    proxy_pass http://127.0.0.1:8000;
    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;
    proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
}
nginx

И учим slowapi читать заголовок

Python
def real_ip(request: Request) -> str: return request.headers.get( "x-real-ip", request.client.host, ) limiter = Limiter(key_func=real_ip)
13 · Logrotate

Режем логи каждый день, храним 7 дней

/etc/logrotate.d/vibe-api

/var/log/vibe-api.log {
    daily
    rotate 7
    compress
    delaycompress
    missingok
    notifempty
    create 0640 ubuntu ubuntu
    postrotate
        systemctl reload vibe-api > /dev/null 2>&1 || true
    endscript
}
bash

Что делает каждая строка

  • daily — ротация раз в сутки
  • rotate 7 — хранить 7 архивов, потом удалять
  • compress — сжимать в .gz (экономия 10×)
  • delaycompress — вчерашний не жать, чтобы tail не ломался
  • missingok — не падать, если файла нет
  • notifempty — не ротировать пустой
  • create — создать новый с правами ubuntu
  • postrotate — перезагрузить сервис после
Проверка вручную: sudo logrotate -f /etc/logrotate.d/vibe-api — принудительная ротация. Должен появиться файл vibe-api.log.1.
14 · Мониторинг

Три команды, которые закрывают 90% админских вопросов

htop
Живая картина процессов. CPU, RAM, uptime. F9 — kill, F6 — сортировка. Твой Task Manager для Linux.
df -h
Сколько места на дисках, в читаемом виде (M/G). Если / > 85% — пора чистить логи.
free -h
Сколько RAM занято. Если available близко к нулю — OOM Killer скоро убьёт процесс.
ss -tlnp
Кто какой порт слушает. Новый netstat. Увидел незнакомый LISTEN на 8000 — проверь.
journalctl -u vibe-api -f
Логи systemd-сервиса в реальном времени. -n 50 — последние 50 строк.
tail -f /var/log/vibe-api.log
Живой tail application-лога. Параллельно с journalctl — видишь и systemd, и приложение.
uptime
Сколько VM живёт + load average. 0.2 0.1 0.05 — покой. 4.0 3.5 2.0 — перегрев.

Домашнее задание (≤ 20 минут)

Задача

Запустить на VM нагрузку: 20 запросов в секунду в течение минуты. Прислать скриншот лога, где видны срабатывания rate-limit.

Скрипт нагрузки

Python
import httpx, asyncio, time async def bomb(url, rps=20, duration=60): delay = 1 / rps end = time.time() + duration async with httpx.AsyncClient() as c: while time.time() < end: asyncio.create_task(c.get(url)) await asyncio.sleep(delay) asyncio.run(bomb("https://твой-домен.ru/horoscope?sign=lev"))

Что прислать

  • Скриншот tail -n 50 /var/log/vibe-api.log
  • В логе должно быть минимум 10 строк с 429
  • Подпись: сколько у тебя получилось OK / 429 за минуту
Запускай тест из другой сети (например, с телефона через hotspot). Если запустишь прямо на VM — всё будет с 127.0.0.1, и rate-limit нечестный.

18 · Грабли

Типичные сложности и лечение

slowapi ругается на request.client.host
За Nginx не видно реальный IP. Прописать proxy_set_header X-Real-IP $remote_addr; и взять его в коде (слайд 13).
Логи не пишутся в файл
Права. sudo touch /var/log/vibe-api.log, sudo chown ubuntu /var/log/vibe-api.log, рестарт сервиса.
429 не возвращается, все запросы идут
Забыл app.state.limiter = limiter. Классика. Пропущенная строка — пропущенный лимит.
logrotate не ротирует
Проверь sudo logrotate -d /etc/logrotate.d/vibe-api — dry-run покажет, чего не хватает. Часто — прав или missingok.
После ротации в файл ничего не пишется
Без postrotate сервис пишет в заархивированный. Надо дать сигнал — systemctl reload.
Бонус для опережающих
Поставь алёрт в Telegram: systemctl is-active vibe-api через cron раз в минуту — упал сервис = бот шлёт в личку.
